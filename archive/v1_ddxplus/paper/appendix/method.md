# Method Section - Detailed Draft

## 3. Method

### 3.1 Problem Formulation

We formulate automatic differential diagnosis as a sequential decision-making problem. Given a patient's demographics (age, sex) and chief complaint, the system iteratively asks about symptoms until sufficient information is gathered to produce a ranked list of candidate diagnoses.

Formally, let:
- $\mathbf{x} = (a, s, c)$ denote patient demographics: age $a$, sex $s$, and chief complaint $c$
- $\mathbf{e}^+ \subseteq \mathcal{S}$ denote confirmed symptoms
- $\mathbf{e}^- \subseteq \mathcal{S}$ denote denied symptoms
- $\mathcal{D}$ denote the set of possible diagnoses

At each turn $t$, the system observes the current state $(\mathbf{x}, \mathbf{e}^+_t, \mathbf{e}^-_t)$ and either:
1. **Inquire**: Select a symptom $s_t \in \mathcal{S} \setminus (\mathbf{e}^+_t \cup \mathbf{e}^-_t)$ to ask about
2. **Diagnose**: Output a ranked list of diagnoses $\hat{\mathbf{d}} = (d_1, d_2, \ldots, d_k)$

The objective is to maximize diagnostic accuracy while minimizing the number of questions (interaction length, IL).

### 3.2 System Architecture: KG-Guided 2-Hop Traversal

The core contribution of our approach is **KG-guided 2-hop traversal** for candidate generation. Unlike LLM-only approaches that rely on parametric knowledge, we leverage the structured relationships in UMLS Knowledge Graph to systematically explore the symptom-disease space.

Our system consists of two components:

1. **UMLS Knowledge Graph Module (Core)**: Generates and ranks candidates through 2-hop graph traversal
2. **Small LLM Selection Module (Auxiliary)**: Selects among top-K candidates; also serves to validate whether small LLMs possess medical diagnostic capability

The KG handles the heavy lifting of medical knowledge retrieval, while the LLM provides contextual selection from a constrained candidate set.

#### 3.2.1 UMLS Knowledge Graph

We utilize a subset of the Unified Medical Language System (UMLS) Metathesaurus, focusing on:
- **Concept Unique Identifiers (CUIs)**: Standardized medical concept codes
- **Semantic Types**: Disease, Symptom, Finding, etc.
- **Relationships**: may_cause, associated_with, has_finding

The KG is stored in Neo4j graph database for efficient traversal.

#### 3.2.2 Core Method: 2-Hop Traversal for Candidate Generation

The key insight is that differential diagnosis can be formulated as a graph traversal problem. Given confirmed symptoms, we find candidate symptoms through **2-hop traversal**:

```
Symptom → Disease → Symptom (2-hop)
   e⁺   →    d    →   s_candidate
```

Formally, given confirmed symptoms $\mathbf{e}^+$ and chief complaint $c$:

**Step 1 (First Hop):** Find diseases connected to confirmed symptoms
$$
\mathcal{D}_{\text{active}} = \{ d \in \mathcal{D} : \exists e \in \mathbf{e}^+, (e \rightarrow d) \}
$$

**Step 2 (Second Hop):** Find symptoms connected to active diseases
$$
\mathcal{C}_s = \{ s \in \mathcal{S} : \exists d \in \mathcal{D}_{\text{active}}, (d \rightarrow s) \} \setminus (\mathbf{e}^+ \cup \mathbf{e}^-)
$$

**Candidate Scoring:** Each candidate symptom $s$ is scored by **disease coverage**—the number of active diseases it can differentiate:

$$
\text{coverage}(s) = |\{ d \in \mathcal{D}_{\text{active}} : (d \rightarrow s) \}|
$$

This 2-hop traversal ensures:
1. **Systematic exploration**: All medically relevant symptoms are considered
2. **Evidence-based ranking**: Symptoms are ranked by their discriminative power
3. **No hallucination**: Candidates are grounded in UMLS relationships

#### 3.2.3 Diagnosis Scoring

We score each candidate diagnosis $d$ using a coverage-based formula:

$$
\text{score}(d) = \frac{|\mathbf{e}^+ \cap \text{symptoms}(d)|}{|\text{symptoms}(d)| + 1} \times |\mathbf{e}^+ \cap \text{symptoms}(d)|
$$

This formula balances:
- **Precision**: Fraction of disease symptoms that are confirmed
- **Support**: Number of confirmed symptoms

The final probability is obtained by normalizing scores:

$$
P(d | \mathbf{e}^+, \mathbf{e}^-) = \frac{\text{score}(d)}{\sum_{d' \in \mathcal{D}} \text{score}(d')}
$$

#### 3.2.4 Stop Decision

The system decides to stop inquiring and make a diagnosis when any of the following conditions is met:

1. **High confidence**: $P(d_1) \geq 0.8$
2. **Clear separation**: $P(d_1) - P(d_2) \geq 0.3$
3. **Maximum questions**: $t \geq 50$
4. **Minimum questions**: $t \geq 3$ (safety constraint)

### 3.3 LLM Selection Module (Auxiliary)

The LLM serves an **auxiliary role**: selecting among the top-K candidates generated by KG 2-hop traversal. Since the KG already provides high-quality, ranked candidates, the LLM's task is simplified to choosing from a constrained set.

#### 3.3.1 Role of LLM Reasoning

We include a brief reasoning step in the LLM prompt for two purposes:

1. **Capability Verification**: The reasoning output allows us to assess whether small LLMs (1B-8B) possess genuine medical diagnostic reasoning, or merely perform random selection
2. **Interpretability**: Provides a trace of the model's decision process

**Key Finding**: When Top-N increases from 2 to 10, if GTPA@1 remains significantly above random baseline (10%), it indicates the LLM has medical reasoning capability. Our experiments show small LLMs maintain 80%+ accuracy even at Top-10, demonstrating genuine diagnostic capability.

#### 3.3.2 Two-Stage Output Generation

To ensure reliable output from small LLMs, we use a two-stage approach:

**Stage 1 - Brief Reasoning (Unconstrained):**
```
Options (ranked by KG diagnostic value):
1. [Symptom A] (45%)
2. [Symptom B] (32%)
3. [Symptom C] (23%)

Brief reason for your selection:
```
The LLM generates a short reasoning (max 100 tokens). This is kept minimal since the primary diagnostic work is done by the KG.

**Stage 2 - Constrained Selection:**
Using regex-constrained decoding via vLLM, we guarantee valid output:
```python
regex_pattern = f"({'|'.join(str(i) for i in range(1, K+1))})"
guided_decoding = StructuredOutputsParams(regex=regex_pattern)
```

#### 3.3.3 Why KG-First, Not LLM-First

| Approach | Candidate Generation | Selection | Issue |
|----------|---------------------|-----------|-------|
| LLM-only | LLM generates | LLM selects | Hallucination, inconsistency |
| **KG-first (Ours)** | **KG 2-hop** | LLM selects | Grounded, systematic |

The KG 2-hop traversal ensures:
- All candidates are medically valid (exist in UMLS)
- Ranking is based on evidence (disease coverage)
- LLM only needs to choose, not generate

#### 3.3.4 Candidate Count Selection

We conducted ablation studies to determine the optimal number of candidates $K$ (Top-N):

| K | GTPA@1 | IL | Random Baseline |
|---|--------|-----|-----------------|
| 2 | ~90% | 19.1 | 50.0% |
| 3 | ~88% | 19.3 | 33.3% |
| 5 | ~85% | 20.8 | 20.0% |
| 10 | ~80% | 22.0 | 10.0% |

The performance degradation with increasing K follows a predictable pattern:
- **K=2-5**: Minimal degradation, LLM consistently selects correct diagnosis
- **K>5**: Gradual decline as task difficulty increases
- **K=10**: Performance still significantly above random baseline (80% vs 10%)

This demonstrates that small LLMs possess genuine medical reasoning capability rather than random selection. The gap between LLM performance and random baseline widens as K increases, validating that the KG 2-hop approach provides high-quality candidates that enable even small LLMs to make accurate selections.

### 3.4 Training-Free Approach

A key advantage of our method is that it requires no task-specific training:

- **Knowledge**: Provided by pre-existing UMLS relationships
- **Reasoning**: Handled by pre-trained LLM capabilities
- **Adaptation**: New diseases/symptoms added by updating KG

This contrasts with RL-based approaches (AARLC) that require extensive environment interaction and reward engineering.

### 3.5 Computational Efficiency

Our approach is computationally efficient:

| Component | Time per Query |
|-----------|----------------|
| KG traversal | ~5ms |
| LLM inference | ~50ms (batched) |
| Total | ~55ms |

Batched inference with vLLM enables processing of 200+ patients/minute on a single GPU.

## Algorithm

```
Algorithm 1: KG 2-Hop Guided Differential Diagnosis

Input: Patient x = (age, sex, chief_complaint)
Output: Ranked diagnosis list d̂

1:  e⁺ ← {chief_complaint}  // confirmed symptoms
2:  e⁻ ← {}                  // denied symptoms
3:  t ← 0                    // turn counter
4:
5:  while not should_stop(e⁺, e⁻, t) do
6:      // ===== CORE: KG 2-Hop Traversal =====
7:      // 1st hop: e⁺ → diseases
8:      D_active ← KG.find_diseases(e⁺)
9:      // 2nd hop: diseases → candidate symptoms
10:     C_s ← KG.find_symptoms(D_active) \ (e⁺ ∪ e⁻)
11:     // Rank by disease coverage
12:     C_s ← sort_by_coverage(C_s, D_active)[:K]
13:
14:     // ===== Auxiliary: LLM Selection =====
15:     prompt ← format_prompt(C_s, x, e⁺, e⁻)
16:     reason ← LLM.generate(prompt, max_tokens=100)  // brief reasoning
17:     s_idx ← LLM.select(prompt + reason, regex="[1-K]")
18:     s* ← C_s[s_idx]
19:
20:     // Patient: Respond
21:     if patient.has_symptom(s*) then
22:         e⁺ ← e⁺ ∪ {s*}
23:     else
24:         e⁻ ← e⁻ ∪ {s*}
25:
26:     t ← t + 1
27: end while
28:
29: // ===== CORE: KG Diagnosis Scoring =====
30: // Score each disease by symptom coverage
31: C_d ← KG.score_diagnoses(e⁺, e⁻)[:K]
32:
33: // ===== Auxiliary: LLM Final Selection =====
34: prompt_d ← format_diagnosis_prompt(C_d, x, e⁺, e⁻)
35: reason_d ← LLM.generate(prompt_d, max_tokens=100)
36: d_idx ← LLM.select(prompt_d + reason_d, regex="[1-K]")
37: d̂ ← C_d[d_idx]
38:
39: return d̂
```

**Key Insight:** The **KG 2-hop traversal** (lines 6-12) is the core contribution—it systematically generates medically valid candidates by traversing symptom→disease→symptom paths. The LLM's role (lines 14-18, 33-37) is auxiliary: selecting from pre-ranked candidates. This design leverages structured medical knowledge from UMLS while using small LLMs only for final selection, achieving high accuracy without large model costs.
